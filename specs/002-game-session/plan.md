# Implementation Plan: Game Session Lifecycle

**Branch**: `002-game-session` | **Date**: 2026-06-20 | **Spec**: `/specs/002-game-session/spec.md`

**Input**: Feature specification from `/specs/002-game-session/spec.md`

## Summary

Implement the foundational game session lifecycle for Daring Cyclops: a server-
managed state machine (WAITING → ACTIVE → ENDED) that supports session creation
with first-player game options, side selection with balance enforcement, a pre-
game lobby with a restricted command set, ship activation, kill-queue respawn
tracking, and session end conditions. All operations are exposed as unary gRPC
RPCs defined in `proto/session/v1/session.proto`. The server is authoritative
for all state transitions. Multiple concurrent sessions are supported and fully
isolated from each other.

## Technical Context

**Language/Version**: Go 1.26.3

**Primary Dependencies**: `google.golang.org/grpc`, `google.golang.org/protobuf`,
`google.golang.org/grpc/test/bufconn` for in-memory integration tests,
`hash/fnv` (stdlib) for tournament seed derivation

**Storage**: In-memory for this feature; the constitution defers persistence
architecture; all session state is lost on server restart

**Testing**: `go test` for unit and integration tests; `bufconn` for in-process
gRPC testing without real sockets

**Target Platform**: Linux server runtime; macOS/Linux developer and CLI runtime

**Project Type**: Separate CLI client and gRPC server binaries in one Go module

**Performance Goals**: All session lifecycle RPCs complete in under 100 ms on a
local or LAN-hosted server; up to 18 concurrent players per session

**Constraints**: gRPC with Protocol Buffers as source of truth; no authentication
for this feature; no persistence; server is authoritative for all session state,
side assignment, and kill-queue management; simulation time governs gameplay
timers, not wall-clock time; `protoc`, `protoc-gen-go`, `protoc-gen-go-grpc`
required for code generation

**Scale/Scope**: Seven unary RPCs, one new CLI sub-command group (`session`),
one shared protobuf contract, full session lifecycle from creation through end,
kill-queue with configurable countdown, balance enforcement, concurrent-session
isolation

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### Pre-Design Gate Review

- **Go client/server shape**: PASS. Separate CLI and server binaries preserved.
- **gRPC and protobuf contract**: PASS. All session operations defined as
  protobuf RPCs in `proto/session/v1/session.proto`.
- **Server authority**: PASS. Server enforces all state transitions, side
  assignments, and kill-queue rules; the client is presentation only.
- **Multiple concurrent game instances**: PASS. Session IDs and per-instance
  isolation are central to the design.
- **Simulation time model**: PASS. Kill-queue countdown is expressed in
  simulation ticks, not wall-clock time.
- **Linux hosting compatibility**: PASS. No OS-specific dependencies.
- **Pluggable persistence**: PASS. In-memory storage behind a clear boundary;
  no persistence coupling introduced.
- **Docker deployment**: PASS. Nothing in the design conflicts with container
  deployment.

### Post-Design Gate Review

- **Interface contract versioning**: PASS. Contract defined under `v1` namespace.
- **Session isolation**: PASS. Each `GameSession` struct is independent; no
  shared mutable state across sessions.
- **Simulation time compliance**: PASS. `respawn_wait_ticks` is a simulation
  tick counter, not a wall-clock duration.
- **Operational reproducibility**: PASS. Validation commands documented in
  `quickstart.md`; tournament seed derivation is deterministic and documented.
